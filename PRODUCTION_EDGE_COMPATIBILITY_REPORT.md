# Diagnostic Report: Edge Browser Compatibility Investigation

**Target URL:** `https://rrs-vhgr.onrender.com/`
**Date:** May 18, 2026
**Status:** ⚠️ Identified Potential Root Causes

## 1. Executive Summary
The investigation into the instability of the Rider Scheduling System on Microsoft Edge revealed several non-standard architectural patterns that likely trigger Edge's aggressive security and efficiency features. While the site works in other browsers, Edge's stricter "Strict Tracking Prevention" and "Efficiency Mode" appear to conflict with the app's current cache-management and session-handling logic.

## 2. Key Diagnostic Findings

### A. Non-Standard Cache/Service Worker Management
In `src/main.tsx`, the application is configured to:
- **Forcefully unregister** all service workers on every load.
- **Delete all browser caches** on every load.

**Impact on Edge:** This behavior is characteristic of scripts attempting to bypass browser security or tracking prevention. Edge's "Strict" mode may flag this as suspicious, leading to the `net::ERR_CONNECTION_RESET` errors observed during testing. This also prevents Edge from utilizing its own "Sleeping Tabs" and "Startup Boost" features effectively.

### B. Render.com "Cold Start" Latency
The application is hosted on Render.com (likely Free Tier). Initial navigation frequently timed out (exceeding 10s).
**Impact on Edge:** Edge's SmartScreen and network stack can be less patient with "stalled" connections compared to Chrome. If the server takes >30 seconds to wake up, Edge may drop the connection entirely to "save resources," leading to the intermittent "sometimes it works" behavior.

### C. Legacy cPanel Technical Interception (Status 428)
The `AuthContext.tsx` contains specialized logic to handle HTTP 428 status codes related to "cPanel Technical Domain interception."
- This logic triggers another full page reload and cache purge.
- If the Cloudflare/Render proxy intermittently forwards headers that trigger this condition, the app enters a reload loop that Edge is likely to kill to prevent resource exhaustion.

### D. Socket.io and Efficiency Mode
The app relies heavily on real-time sockets. 
**Impact on Edge:** Edge's "Efficiency Mode" aggressively throttles tabs that use continuous network resources if they aren't in the immediate foreground. This can cause the "Live" connection to drop or the UI to "freeze," which users perceive as the site not working.

## 3. Root Cause Analysis
The primary reason for Edge's instability is the **Aggressive Client-Side Reset** logic. By clearing all caches and workers on every load, the app forces the browser to re-download 100% of its assets (including large JS bundles and Google Fonts) every time. Combined with Render's cold-start latency, this creates a "heavy" load signature that Edge's efficiency and security layers frequently interrupt.

## 4. Recommended Fixes

### Phase 1: Stabilize Initialization (High Priority)
1. **Remove the aggressive cache purge in `main.tsx`:** Only clear specific app caches if a version mismatch is detected, rather than `caches.delete(name)` for all keys on every load.
2. **Implement Graceful Service Worker Updates:** Instead of forced unregistration, use a standard PWA update flow (e.g., `workbox-window`) to allow users to update when a new version is ready.

### Phase 2: Cleanup Legacy Infrastructure
1. **Remove 428 Status Handling:** Since the site has moved to Render.com, the cPanel-specific 428 interception logic is likely redundant and potentially harmful if triggered by false positives.

### Phase 3: Optimize for Performance
1. **Asset Optimization:** The main JS bundle is being re-downloaded constantly. Fixing the cache logic will allow Edge to use "Startup Boost," significantly improving the "sometimes working" cold-start experience.
2. **Keep-Alive Heartbeat:** Ensure the Socket.io connection uses a lightweight heartbeat to prevent Edge from putting the tab to "sleep" during active sessions.
